## What This Closure Pass Added

The closure pass made four concrete additions relative to the pre-existing implementation:

1. **Unit 1 — `include_control` subscription opt-in and `subscribe_with_control()` method.** Prior to this change, `EventBusClient` subscriptions filtered out `MessageType::Control` envelopes, so `control.collector.lifecycle::BeginMonitoring` and per-collector RPC requests silently dropped at the client side. procmond worked around this by starting collection immediately on startup. The new API lets subscribers opt in (default `false` preserves legacy behavior), and procmond now subscribes to `control.collector.lifecycle` and waits for `BeginMonitoring` before calling `begin_monitoring()`. A new `--standalone` CLI flag preserves agent-less operation for tests and bootstrap. The broker-unreachable path logs a loud WARN and falls back to the escape hatch rather than silently collecting.

2. **Unit 2 — `latency_p99_slo` benchmark with CI gate.** The ticket's "\<1ms p99 latency" acceptance criterion is now a machine-checked assertion in a dedicated criterion bench target that panics if p99 exceeds `Duration::from_millis(1)`. The `eventbus-latency-slo` job in [`.github/workflows/benchmarks.yml`](../../.github/workflows/benchmarks.yml) runs it on Linux and macOS; Windows and FreeBSD are informational per the plan's OS-support posture.

3. **Unit 4 — `e2e_multi_collector` integration test.** Five scenarios exercising the real broker, task distributor, and result aggregator with two simulated collector clients: load balancing, broadcast config reload, failover on deregister, result aggregation with correlation IDs, and wildcard observer. Runs in under a second; scope-limited to in-process clients because cross-OS-process subscription registration is a latent transport-layer gap (see note below).

4. **Unit 3 — doc-truth sync.** `IMPLEMENTATION_SUMMARY.md`, `VALIDATION_CHECKLIST.md`, `COMPREHENSIVE_REVIEW.md`, and `README.md` were updated to match actual behavior. The largest correction was removing the "Pause/Resume stubbed" claim — [`daemoneye-eventbus/src/rpc.rs`](../src/rpc.rs) lines 1862-1950 delegate to `ProcessManager::{pause,resume}_collector`, they are not stubbed. Line-count expectations in `VALIDATION_CHECKLIST.md` were refreshed against current reality. Stale completion-percentage claims were dropped.

## Known Latent Gaps (Not Blocking Closure)

These were surfaced during the closure pass but are explicitly out of scope for END-297 and tracked separately:

- **Cross-OS-process subscription registration over `interprocess`** (scope-reducing R14 from full to partial — see matrix row 14). The broker does not currently run a per-connection receive loop that handles subscribe-request protocol messages from remote clients. `ClientConnectionManager::subscribe_client` exists but is only exercised by an in-process test in [`daemoneye-eventbus/src/transport.rs`](../src/transport.rs). This means the `e2e_multi_collector` test scenarios deliberately use in-process clients. The broker's routing, load balancing, queue groups, and failover logic are all exercised through the real code paths, but genuinely separate OS processes subscribing via Unix socket is a follow-on transport-layer item. **In production today, a separately-spawned procmond will hit its 60s `BEGIN_MONITORING_WAIT_TIMEOUT` and fall back to standalone collection on every startup until this gap closes.**
- **File-size hygiene.** [`daemoneye-eventbus/src/rpc.rs`](../src/rpc.rs) (2789 lines) and [`daemoneye-eventbus/src/broker.rs`](../src/broker.rs) (1371 lines) exceed the AGENTS.md 500-600 line guideline. A separate refactor ticket is the right home for that.
- **`SECURITY_TODO` at [`daemoneye-eventbus/src/task_distribution.rs:149`](../src/task_distribution.rs).** The `CollectorStatus::ShuttingDown` variant is reserved for Task 2.5.5 graceful-shutdown coordination and is marked `#[allow(dead_code)]` intentionally.
- **Coverage metrics.** `cargo llvm-cov` is not yet gated in CI. [`COMPREHENSIVE_REVIEW.md`](../COMPREHENSIVE_REVIEW.md) tracks this as a medium-priority follow-up.

## Post-Merge Follow-ups

Surfaced by the final comprehensive review (6 reviewers), deferred to focused follow-up PRs rather than expanding this already-large closure pass:

- **Audit-ledger writes on fallback paths** (P1, security-relevant). Every fallback in `procmond/src/main.rs` (operator-opt-in standalone, registration failure, broker-unreachable subscribe fail, lifecycle-wait timeout, lifecycle-wait channel closed) emits a `warn!`/`error!` log but does not write to the BLAKE3 hash-chained audit ledger. An attacker who disables the agent can force uncoordinated collection with no tamper-evident record. Work required: decide on signing-key scope for procmond, initialize the `AuditLedger` (see `daemoneye-lib/src/crypto.rs:172+`) in procmond startup, wire signed entries at the five transition sites, add an integration test.

- **Test coverage for new procmond startup flow** (P1). Four specific paths still untested at HEAD: subscribe-then-wait happy path (no workspace-level test delivers `BeginMonitoring` through the broker to a subscriber and asserts the Running transition); `--standalone` / `PROCMOND_STANDALONE=1` runtime behavior (only CLI-help snapshot exists); `lifecycle_wait_task` non-lifecycle continue branch; `TrySendError::Full` backpressure drop in `handle_control_message_internal`. Combined regression surface is narrow but real.

- **Remove `EventBusConnector.connected` bool** (P2). Duplicates `self.client.is_some()` and can drift — `try_reconnect` error paths at lines 685 and 861 set `connected = false` but leave `client = Some(...)`. Deferred from this PR because alignment requires auditing each error path to decide whether to drop the client there.

- **`#[non_exhaustive]` on `EventSubscription`** (P2). Attempted in this PR; requires a builder or `with_*` setter API because cross-crate struct-literal construction (including FRU) is forbidden for non-exhaustive types, which would cascade through `collector-core`, tests, and benches. Docstring at [`daemoneye-eventbus/src/message.rs`](../src/message.rs) notes the deferral.

- **Type-design polish** (P3). Collapse `EventSubscription::include_control` into the method choice (`subscribe()` vs `subscribe_with_control()`); upgrade `EventBusClient::shutdown_signal(&self) -> bool` to a three-variant enum (`Sent` / `AlreadyShutDown` / `NoReceivers`); either reject `include_control=false` from `subscribe_with_control()` or change return to `(Receiver<BusEvent>, Option<Receiver<Message>>)`.

- **Simplification opportunities** (P3). Extract `try_subscribe_control` helper in `procmond/src/main.rs` to flatten the nested match at lines 476-538; consolidate the channel-closed and timeout fallback arms in `lifecycle_wait_task` (now-shared `begin_monitoring_or_exit` helper already does this at the call-site level); add partial-count warning to `collect_until` rustdoc; promote the `Arc::into_inner` shutdown pattern to a canonical example on `shutdown_signal`'s rustdoc; pull `DEFAULT_SUBSCRIPTION_CAPACITY = 1000` into a named `const`.
